# 从编写Loader的角度分析Loader机制

# 编写属于自己的 Loader

Loader 是 Webpack 的核心机制，但是内部的工作原理其实非常简单。接下来我们一起来开发一个自己的 Loader，通过这个开发过程再来深入了解 Loader 的工作原理

这里我们编写一个 markdown loader 为 🌰 ，以便可以在代码中直接导入 md 文件。我们都应该知道 markdown 一般是需要转换为 html 之后再呈现到页面上的，所以我希望导入 md 文件后，直接得到 markdown 转换后的 html 字符串，如下图所示：

// ./md-loader.js
const loaderUtils = require("loader-utils");
const marked = require("marked");

module.exports = function(source) {
  const options = loaderUtils.getOptions(this);
  console.log(source, options);
  let html = marked(source);
  console.log(html);
  return html;
};

这里有一个注意点，因为 loader 内部经常用到 this，我们不能写成箭头函数。

如果内部需要做异步的操作，这时候要用到 async 函数注册回调函数 查阅官方 API。 代码要这么写

const loaderUtils = require("loader-utils");
const marked = require("marked");
module.exports = function(source) {
  const options = loaderUtils.getOptions(this);
  const callback = this.async();

  let html = null;
  setTimeout(() => {
    html = marked(source);
    callback(html);
  }, 1000);
};

这里我们使用来 marked 这个库来实现 markdown 语法到 html 的转换。完成以后，我们回到 Webpack 配置文件中添加一个加载器规则，这里匹配到的扩展名是 .md，使用的加载器就是我们刚刚编写的这个 markdown-loader.js 模块，具体代码如下所示：

// ./webpack.config.js
module.exports = {
  entry: './src/main.js',
  output: {
    filename: 'bundle.js'
  },
  module: {
    rules: [
      {
        test: /\.md$/,
        // 直接使用相对路径
        use: './loaders/md-loader'
      }
    ]
  }

控制台提示说我们需要另外一个合适的 loader 来接受这个结果。这是因为我们在 md-loader 里面给 webpack 返回的其实是一些 html 的字符串，Webpack 内部默认只认 js 不认其他的内容，所以我们有两种方式来解决这个问题：

  1. 将最终的 html 字符串包装成为 JavaScript 代码返回

  2. 引入另一个 loader 进行处理

# 总结 Loader 的运行机制

Loader其实就是一个被导出的函数，在webpack.config.js文件中配置后，Webpack会在运行时解析到特殊模块后，将loader加载并执行，执行结果依次传递至下一个loader，直到返回给Webpack

Loader 是 Webpack 模块化思想的核心的机制，因为正是有了 Loader 机制，Webpack 才能足以支撑整个前端项目模块化的大梁，实现通过 Webpack 去加载任何你想要加载的资源。
